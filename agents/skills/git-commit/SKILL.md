# Git Commit Skill

分析当前项目的 git 改动，生成符合约定式提交规范的 commit message 并执行提交。

## 执行步骤

### 1. 检查 git 状态
运行以下命令了解当前改动：
```bash
git status --short
git diff --cached  # 已暂存的改动
git diff           # 未暂存的改动
```

### 2. 分析改动内容
根据 diff 内容判断改动类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | 添加任务完成进度条 |
| `fix` | 修复 bug | 修复日期选择器时区问题 |
| `refactor` | 代码重构 | 重构任务列表组件 |
| `docs` | 文档更新 | 更新 API 文档 |
| `style` | 代码格式 | 调整代码缩进 |
| `test` | 测试相关 | 添加单元测试 |
| `chore` | 构建/工具 | 更新依赖版本 |

### 3. 生成 commit message
遵循以下格式：
```
<type>: <简短描述>

<详细描述（可选）>

<关联 issue（可选）>
```

**规则：**
- 简短描述不超过 50 个字符
- 使用中文描述，简洁明了
- 详细描述每行不超过 72 个字符
- 如果是破坏性变更，在类型后加 `!`，如 `feat!:`

### 4. 用户确认
向用户展示生成的 message，询问：
- 是否直接提交？
- 是否需要修改 message？
- 是否需要先 `git add` 未暂存的文件？

### 5. 执行提交
根据用户选择执行：
```bash
# 如果用户确认
git commit -m "<message>"

# 如果用户想修改，使用编辑器
git commit  # 打开编辑器
```

## 特殊情况处理

- **无改动**：提示用户没有需要提交的改动
- **已有暂存文件**：优先基于暂存内容生成 message
- **部分文件未暂存**：询问是否一并添加
- **提交失败**（如 hook 报错）：显示错误信息，让用户处理
